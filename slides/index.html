The HTML Plan

Advisory Board Edition

Robin Berjon / @robinberjon / http://berjon.com/

The Gory Details

This presentation focuses on participation and the WHATWG.

Enquiring minds should feel free to consult the full plan.

WHATWG Relationship

Cross-organisation discussions about process are tedious, invidious, and have an observed low rate of success.

The core of the policy is therefore rooted in action and being a good citizen while minding one’s own business as a group.

If the participation policy based on CGs succeeds, maybe a situation could arise in which the WHATWG would become one such CG. But that is at best long-term.

WHATWG: HTML

Interference with the WHATWG is kept to a minimum by:

  • Automatic publishing (no cherry-picking)
  • Deforking where possible
  • Aiming to publish a strict subset
  • Moving additive work to other documents

We could reach a point at which the core HTML specification is emptied and only contained in modules. But this is at best far away.

WHATWG: DOM

The DOM specification is already under the automatic publishing and strict subset policy that is suggested for HTML. As far as it reaches, it is a qualified success.

As detailed in the full plan however, taking it further to Recommendation is blocking on some bugs. These are orthogonal to the WHATWG relationship however.

WHATWG: Other documents

The HTML WG does not expect to fork other documents. URL is getting its own group.

Participation: Problems

The HTML WG is seeing abysmal participation.

  • The core parts of HTML are not a priority for many
  • The large group and document are daunting
  • The bad history still scares people away

Part°: (Qualified) Success

We have seen some qualified successes with participation broken up into smaller, more focused groups.

Questions as to whether that constitutes sufficient data to build a decision upon, operating with smaller focused groups makes intuitive sense.

Participation: Structure

Work is distributed to smaller groups, likely Community Groups (but exact modalities are up for discussion). The HTML WG is used as a coordination platform amongst them.

CGs can choose their work mode, tooling, and licenses. However some specifics are encouraged (WebApps work mode, WebSpecs tooling, W3C Software License). Experimentation is welcome as it can inform us all quickly, with minimal discursive overhead.

Participation: Tooling (1)

CGs are encouraged to use GitHub. A tool for full-fledged backups will be needed.

The core group will be using Echidna.

Participation: Tooling (2)

All groups are welcome to use Discourse for general-purpose input from the community, or for full-fledged discussions (as used by asm.js).

CGs (which can’t use Echidna) are encouraged to use the WebSpecs project which provides automated publishing tooling at https://specs.webplatform.org/.
